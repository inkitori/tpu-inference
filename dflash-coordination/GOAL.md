# GOAL (READ-ONLY) — DFlash spec-decode bringup for gpt-oss-20b on TPU v6e-8

> This file is the north star. Every agent manager MUST read it first and must NOT
> drift from it. Do not edit this file.

## Mission
Make `openai/gpt-oss-20b` (TARGET, already supported) + `z-lab/gpt-oss-20b-DFlash`
(DRAFT) speculative decoding work on this TPU v6e-8 Trillium VM, at **concurrency 32**,
on the **torchax** path.

## Hard requirements (ALL must hold to declare DONE)
1. **CORRECTNESS / LOSSLESS:** With greedy/deterministic sampling, target+draft must
   produce the SAME tokens as target-alone — only faster. Exact logit identity is NOT
   required (kernels differ). If tokens differ, you MUST prove it is a genuine bf16
   near-tie (top-2 logits within bf16 epsilon), NOT a verification/alignment bug. The
   accept/reject method itself must be provably correct.
2. **SPEED:** DFlash must be FASTER than target-only at **input=1, output=4096,
   concurrency=32** (end-to-end benchmark on THIS VM, warm XLA cache).
3. **PATH:** torchax everywhere. Target `MODEL_IMPL_TYPE=vllm`,
   draft `DRAFT_MODEL_IMPL_TYPE=vllm`. NOT the jax path.

## Constraints / how to work
- DFlash landed upstream in tpu-inference PRs **#1868 / #1869 / #1870**. Consult these
  FIRST for any spec-decode question or regression.
- Any GPU/CUDA-only op (or op torchax can't trace) surfacing in a TPU trace must be
  ported to a JAX/torchax lowering matching the pure-Torch reference numerics exactly.
- **Commit + push often** (preemptible node — work can vanish anytime).
- Local env: `~/tpu-tooling/tpu-env.sh` wraps the venv + TPU env. HF cache is the

- Relevant skills: `tpu-model-workflow` (correctness/bench discipline on torchax path),

- **SCOPE GUARD (user directive):** Only fix DFlash / spec-decode-SPECIFIC problems. If an issue also
  reproduces on plain `gpt-oss-20b` WITHOUT DFlash (i.e. it is not spec-decode-specific), it is OUT OF
  SCOPE — note it and move on; do NOT go fix general gpt-oss issues. Example: the spec-on `logprobs`
  OverflowError — if it's a general gpt-oss/logprobs issue, leave it; the workaround is to probe
  without logprobs. Stay focused on getting DFlash working.

## Correctness test ladder (use these, cheapest first)
1. **PERFECT-DRAFT MACHINERY TEST (run FIRST):** inject the target's own greedy tokens
   as the "draft proposals" → acceptance MUST be ~100%. If <100% beyond FP near-ties,
   the bug is in the verification/alignment path (off-by-one in positions/seq_lens,
   comparing draft[i] vs target argmax[i+1], etc.), NOT in DFlash.
2. **OFFLINE RE-DERIVE:** dump target logits at verify positions + draft tokens + the
   kernel's accept mask; recompute the accept/reject rule in numpy; assert match.
3. **GREEDY DIVERGENCE POINT:** count tokens agreeing with target-only before first
   divergence. Diverge at step 1-2 ⇒ real bug. Agree dozens then split on near-tie ⇒
   FP-fine. Check first decode step argmax matches exactly (identical context).
4. **DISTRIBUTIONAL (temp>0):** KL / chi-squared over first-token dist vs target-only.
   Final confirmation only — expensive + noisy, not a debugging tool.

## Definition of DONE (Phase 1)

and a final report file written. Only then is Phase 1 done.
