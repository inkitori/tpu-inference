## 2. The v6e-16 size constraint — what it does and does NOT block

- **It does NOT block the plan.** Every test in Phases 0–2 uses a **tiny random-weight `GlmMoeDsaConfig`** (hidden 64, 4 layers, 4 experts). Model size is irrelevant there.
- **It DOES block loading the real GLM-5.2 checkpoint.** Real GLM 5.2 (78 layers, hidden 6144, 256 experts) is ~hundreds of GB in bf16 — it will **not** fit in v6e-16 HBM (~512 GB total, and you'd need room for activations/optimizer-free inference still won't fit a >700GB bf16 model). **Do NOT attempt to download or load the real checkpoint on this box.** All correctness work is against the HF eager oracle on the tiny config.
- For TPU runs, use the **tiny config** on 1 chip (single-device mesh) and, where the plan/spec call for sharding, the tiny config across the 16 chips. That is enough to validate the kernel + sharding logic that the CPU box could not.
