# GLM 5.2 on JAX/TPU — Research Findings

**Date:** 2026-06-19
**Purpose:** Distilled output of a 5-agent research sweep, to ground the design/implementation plan for bringing up **GLM 5.2** as a native JAX model on TPU in `tpu-inference`, validated for mathematical correctness on CPU first.
**Status:** Research complete except MTP-head structure + exact GLM-5.2 hyperparameter table (pending final sub-agent; marked TODO below).

---

## TL;DR / Key conclusions

1. **GLM 5.2 ≈ DeepSeek-V3 architecture + DeepSeek Sparse Attention (DSA).** MLA attention + fine-grained MoE (sigmoid, group-limited `noaux_tc` routing, shared experts) + a **lightning-indexer sparse-attention** path. It is *very* close to the existing `tpu_inference/models/jax/deepseek_v3.py`.
2. **GLM's DSA indexer is the simpler V3.2 flavor** (raw weighted dot-product → top-2048, **no softmax, no ReLU**, `k_norm`=LayerNorm), **unlike** the `../claude-deepseek-v4` fork's V4 indexer (ReLU + compressor/CSA). So we *adapt* the fork, we don't copy it.
3. **GLM has zero native-JAX footprint today** — it currently only runs via the vLLM-PyTorch fallback path. A native JAX impl must be built + registered.
4. **The bulk of the model is reusable** from `deepseek_v3.py` (MLA stack, MoE, router, RoPE, weight loader, sharding). The genuinely *new* code is: (a) config-driven hparams, (b) the DSA lightning indexer module + top-k selection, (c) a sparse-attention kernel/path, (d) optionally MTP.
5. **The "breaks on multiple batches" issue in the fork is a MoE/runner bug, NOT a DSA bug.** DSA itself is correct and CPU-validated in the fork.
6. **CPU correctness is achievable but requires writing pure-`jnp` reference paths** — the production MLA Pallas kernel has no CPU/interpret mode. The fork already has a pure-JAX DSA reference we can mirror. Mac install needs to drop the `libtpu` pin and the AVX2 XLA flag.

## 1. What GLM 5.2 is (architecture spec) — *vLLM `glm4_moe*` reference*

**Family:** GLM MoE (`vllm/model_executor/models/glm4_moe.py` + `glm4_moe_mtp.py`, `glm4_moe_lite*`). DeepSeek-V3-shaped.

**MLA attention:**
- Fused QKV-a down-projection; `q_a` and `kv_a` RMSNorms (LoRA-rank norms).
- RoPE applied **only to the rope slice** (`qk_rope_head_dim`); **interleaved/GPT-J style for MLA**, **NeoX style for the indexer**.
- `softmax_scale = qk_head_dim^-0.5 · mscale²` (YaRN mscale).
- Non-absorbed reference path is acceptable for CPU correctness.

**DSA lightning indexer (the novel piece):**
- Separate **32-head** MQA scorer (config=32 for real GLM-5; code default 64) reading the shared `q_c` latent.
- `k_norm` = **LayerNorm (eps 1e-6), NOT RMSNorm**.
- RoPE on the first 64 dims (NeoX).
- **Score = raw weighted dot-product → top-2048; NO softmax, NO ReLU** (DeepSeek-V3.2 form).
- Per-layer sparsity schedule: `index_skip_topk_offset=2`, `index_topk_freq=1` ⇒ **first ~2 layers dense, the rest sparse**.
- Config flag that turns DSA on: presence of `index_topk` (V3.2 style). (V4/CSA style uses `compress_ratio==4` + a compressor — *not* GLM 5.2.)

**MoE / router:** sigmoid scoring, `e_score_correction_bias`, group-limited `noaux_tc` top-k (`n_group`/`topk_group`), `routed_scaling_factor`, `norm_topk_prob`, `first_k_dense_replace` leading dense layers, `n_shared_experts`. Matches DeepSeek-V3 router exactly in shape.

**MTP:** `glm4_moe_mtp.py` provides a multi-token-prediction head (`num_nextn_predict_layers`). *(Exact structure: TODO — pending sub-agent.)*

**Exact hyperparameters (num_hidden_layers, hidden_size, intermediate_size, vocab_size, lora ranks, head dims, n_group/topk_group, n_routed/shared_experts, a small/"lite"/"Air" variant for CPU testing):** **TODO — pending final sub-agent**; use `deepseek_v3.py:76-108` defaults as the reference shape until confirmed.

## 2. DSA algorithm + the fork's implementation — *`../claude-deepseek-v4` (a DeepSeek-V4 impl)*

**Algorithm (precise):**
- **Lightning indexer:** cheap auxiliary scorer ranking past tokens per query. q from q-LoRA compressed vector; k from hidden states; per-head weights; `k_norm`; `softmax_scale=head_dim^-0.5`. RoPE on the rope slice via a *separate* indexer rope.
  - vLLM V3.2 (= GLM target): `score = weighted MQA dot-product`, **no ReLU/softmax**.
  - Fork V4: `index_score = relu(einsum("bshd,btd->bsht", q, kv)) * weights`, sum over heads → `[B,S,T]` (`deepseek_v4_attention.py:478-485`). **GLM does NOT use the ReLU.**
- **Top-k:** `index_topk` (GLM=2048). `lax.top_k(score, K)`; causal/invalid positions set to `-1` sentinel.
- **Sparse MLA:** attends only over selected KV rows (gather, not mask, in production; equivalent to mask-to-`-inf` for testing). Per-head **attention sink** contributes to the softmax denominator only.
- **Paged cache:** indexer K kept in its own (fp8) paged cache; prefill chunked, decode reads paged via block-table. *The fork deliberately bypasses ragged-paged-attention and uses a contiguous per-request KV — works at `MAX_SEQS=1`.*

**Fork implementation status (authoritative: `HANDOFF_PERF.md`, `HANDOFF_QUANT.md`, ~2026-05-29):**
- **Working & CPU-validated** (bit-parity vs torch): plain-JAX lightning indexer (`deepseek_v4_attention.py:434-499`), `lax.top_k` selection, Mosaic `sparse_attn` kernel (`kernels/sparse_attn/kernel.py:117`), AND a **pure-JAX `sparse_attn` reference auto-selected when there is no mesh** (`deepseek_v4_attention.py:179-216`) — **this is the CPU correctness door.** Kernel wiring is live (`:893` decode, `:986` prefill); its own "NOT yet wired" docstring is stale.
- **Perf:** 146 ms/step ≈ 0.43 tok/s @ N=1. Bottleneck = **MoE FP4→bf16 dequant (105 ms)** + launch-bound tiny tiles, **not** the sparse path (sparse kernel is optimal, 5× faster than pure-JAX).
- **Multi-batch = confirmed silent corruption, but in the MoE/runner path** (`_v4_decode_replicate` gated `==1`; `num_reqs>1` → token-sharded activation → `N(4)<16` forces a dense MoE einsum that reads **uninitialised HBM** for un-owned tokens). **Not a DSA bug.** Multi-seq dispatch is also still a serial Python loop.
- **Reuse as-is:** Mosaic sparse-attn kernel, the indexer, the CPU oracle/microbench harness, the pure-JAX reference. **Rework needed:** concurrent decode / multi-batch MoE, ragged-batch dispatch, MTP spec-decode.

**CPU reference path (mathematically equivalent, plain JAX):**
1. Indexer: `weights = x @ weights_proj.T · softmax_scale · H^-0.5`; `index_score = einsum("bshd,btd->bsht", q, kv)` (GLM: **no ReLU**) `· weights`; sum over heads → `[B,S,T]`.
2. Mask causal `-inf`; `_, idx = lax.top_k(score, K)`; invalid → `-1`.
3. Attention: `logits = einsum("bmhd,bmkd->bmhk", q, kv)·scale`; non-top-k cols → `-inf`; softmax with sink in the denominator; `out = einsum("bmhk,bmkd->bmhd", p, kv)`. (Mask-all-N variant is simpler and bit-equivalent for tests.)

## 3. tpu-inference model-definition layer + GLM delta — *`deepseek_v3.py` template*

**Module system: Flax NNX** (never linen). Base `JaxModule(nnx.Module)` (`layers/jax/__init__.py:20`) adds torch-style `named_parameters`/`named_children` so vLLM's `AutoWeightsLoader` works against JAX params. Modules are `@dataclass(kw_only=True)` (with `__post_init__(rngs)`) or plain `__init__`. Primitives: `JaxEinsum(nnx.Einsum)` (`linear.py:29`), `JaxRmsNorm(nnx.RMSNorm)` (`norm.py:25`). Params via `nnx.Param`/`create_param`/`nnx.with_partitioning(init_fn, spec)`. **No jit on the class** — dispatch is external via the frozen-dataclass `ModelInterface` (`interface.py:40`); the class exposes `__call__`, `compute_logits`, `load_weights`.

**MoE:** `DeepseekV2Moe` (`:840`) = `DeepSeekV3Router` (`:983`) + `shared_experts` (`DeepseekV3MLP`) + `SharedFusedMoe(JaxMoE)` (`:815`). ## 6. Synthesis — recommended approach

**GLM 5.2 native-JAX bring-up = `deepseek_v3.py` + 3 new pieces, correctness-first on CPU.** Recommended phasing:

- **Phase 0 — CPU dev env:** working JAX-CPU install on the Mac (no libtpu/AVX2), import-clean, a tiny GLM config + a pure-`jnp` reference scaffold + an HF-transformers-eager oracle.
- **Phase 1 — Dense GLM (no DSA):** new config-driven `glm_moe.py` adapting `deepseek_v3.py` (MLA + MoE + router + QK-norm). Validate per-submodule on CPU vs HF-eager (router, norms, MoE-with-`USE_DENSE_MOE`, MLA via a pure-`jnp` MLA reference). This de-risks ~80% of the model independent of DSA.
- **Phase 2 — DSA:** add the lightning-indexer module + top-k selection (V3.2 form: no ReLU, LayerNorm k_norm, top-2048, first ~2 layers dense). Validate the indexer + sparse attention on CPU against a pure-`jnp` reference (mirroring the fork's `sparse_attn` reference) and against full-MLA where mathematically expected. Port/adapt the fork's Mosaic `sparse_attn` kernel for the TPU path.
- **Phase 3 — TPU bring-up:** wire the real Pallas/Mosaic kernels, paged-KV integration, sharding; single-sequence correctness on TPU; then performance + multi-batch (the fork's known MoE multi-batch bug must be solved here).
- **Phase 4 (optional) — MTP** for speculative decode.

**Reuse map:** MLA stack / MoE / router / RoPE / weight loader / sharding from `deepseek_v3.py`; DSA algorithm + Mosaic kernel + pure-JAX reference from the `../claude-deepseek-v4` fork (adapted V4→V3.2 indexer).
