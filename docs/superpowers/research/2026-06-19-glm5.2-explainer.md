## 1. What GLM 5.2 actually is (10,000-ft view)

A **decoder-only transformer** (Llama/GPT family) with three non-standard pieces. A vanilla transformer does, per layer: `attention → feed-forward (FFN)`. GLM 5.2 replaces *both* halves with cheaper, cleverer versions:

| Vanilla transformer | GLM 5.2 |
|---|---|
| Multi-head attention (stores full K,V per token) | **MLA** — compresses K,V into a tiny latent vector |
| Attend to **all** past tokens (O(n²)) | **DSA** — attend to only the **top-2048** most relevant past tokens |
| One big FFN per layer | **MoE** — 64 expert FFNs, route each token to 4 |

All three are the same idea in different places: **do less work without changing the answer much.** MLA saves memory, DSA saves attention compute on long contexts, MoE saves FFN compute. Everything else (token embedding → N decoder layers → final RMSNorm → `lm_head` → vocab logits) is a normal transformer.

## 2. The three building blocks (with real GLM 5.2 numbers)

### 2a. MLA — Multi-head Latent Attention (a *memory* optimization)
**Problem:** during generation a transformer caches Key/Value vectors for every past token (the "KV cache"). With 20 heads × 256 dims that's `20×256×2 = 10,240` numbers/token — for a 100k-token context, ~2 GB *per layer*.

**What MLA does:** cache one small shared **latent** and reconstruct per-head K/V on the fly:
- **Down-project** the hidden state to a `kv_lora_rank = 512` latent + a `qk_rope_head_dim = 64` shared rope key. That 576-number vector is the **only** thing cached — ~**18× smaller** than 10,240.
- At attention time, **up-project** (`kv_b_proj`) the 512-latent into per-head `k_nope (192)` and `v (256)` for all 20 heads.
- Query path is also compressed: `q_a_proj → q_lora_rank 768 → q_a_layernorm → q_b_proj →` split into `qk_nope_head_dim 192` + `qk_rope_head_dim 64`.
- **RoPE** (rotary position encoding) applies **only** to the 64-dim rope slice; the 192-dim "nope" slice carries content. Concatenate → 256-dim Q/K per head.

In code you'll see exactly: `q_a_proj → q_a_layernorm → q_b_proj → split nope/rope → apply RoPE`, and `kv_a_proj_with_mqa → split latent/k_rope → kv_a_layernorm → (later) kv_b_proj`. Those two `*_layernorm`s are the **only** norms in attention — **GLM 5.2 has NO per-head QK-norm** (confirmed; means the DeepSeek MLA module transfers essentially unchanged).

### 2b. DSA — DeepSeek Sparse Attention (a *compute* optimization for long context)
**Problem:** even with MLA, *computing* attention is O(n²) — each new token compares against all past tokens.

**What DSA does:** add a cheap "scout" — the **lightning indexer** — that ranks all past tokens and says "these 2048 matter most." The expensive MLA attention then runs **only over those 2048**, regardless of context length.
- A small multi-query projection (`index_n_heads ≈ 64`, but **one shared key head**, so cheap) reads the compressed query latent.
- Computes a **raw weighted dot-product** score vs every past token — **no softmax, no ReLU** (the simple "V3.2" form; the fork we borrow from uses a fancier "V4" form with a ReLU + compressor — GLM does *not*, so we strip those out).
- `k_norm` here is a **LayerNorm (eps 1e-6)**, separate from MLA's RMSNorms.
- Take **`top-2048`** by score (`lax.top_k`), output selected indices, mark the rest `-1`.
- **Not every layer is sparse:** a schedule (`index_skip_topk_offset = 2`) keeps roughly the **first ~2 layers dense** and the **rest sparse**. Early layers benefit from seeing everything; later layers can be selective.

### 2c. MoE + router (a *compute/capacity* optimization)
**Problem:** you want huge capacity but don't want to run all parameters per token.

**What MoE does:** replace the single FFN with **64 expert FFNs** (`moe_intermediate_size = 1536`) + **1 shared expert** (always runs). A **router** picks **top-4** experts/token → compute `4 + 1 = 5` experts instead of 64 (~13× less FFN work) while "containing" 64 experts of knowledge.

**Router** (subtle-bug hotspot, so detail):
1. fp32 linear scores all 64 experts.
2. **sigmoid** (not softmax — GLM/DeepSeek choice).
3. add learned **`e_score_correction_bias`** (load balancing).
4. pick **top-4**. (GLM has `n_group = topk_group = 1` → DeepSeek's group-limited routing is effectively **off** — simpler for us.)
5. **`norm_topk_prob`**: normalize the 4 weights to sum to 1.
6. multiply combined expert output by **`routed_scaling_factor = 1.8`** — applied to the **output**, not the router logits.

Also: the first `first_k_dense_replace` layers use a normal **dense** FFN (intermediate 10240), not MoE.

## 3. What the files/layers actually do (codebase map)

### Three repos
- **`../vllm`** — the **GPU reference** (PyTorch); numerical ground truth we match. Key: `model_executor/models/glm4_moe_lite.py`, `models/deepseek_v2.py` (shared MLA + `Indexer` GLM delegates to), `layers/sparse_attn_indexer.py`. We **read** it; never run it on TPU.
- **`../claude-deepseek-v4`** — a **working JAX fork** that implemented DeepSeek-V4 + DSA on TPU. We **borrow** its sparse-attention kernel and its **pure-JAX reference** for the indexer. Correct at batch-1, slow, has a multi-batch bug — but the *DSA math* is validated.
- **`tpu-inference`** — where we build; the JAX/TPU serving stack.

### Four layers inside `tpu-inference` (top→bottom)
1. **Model definition** — `models/jax/deepseek_v3.py` (template) and the new `models/jax/glm4_moe_lite.py` we write. Pure architecture in Flax **NNX** (NNX = JAX's neural-net lib; a module is a Python object holding params + a `__call__`). Defines attention, MoE, router, decoder layer, full model, and **weight loading** (HF tensor names → our params). ~80% reused from deepseek_v3.py.
2. **Shared layers** — `layers/jax/` (`linear.py`, `norm.py`, `moe.py`, rope, attention modules): reusable bricks. `layers/common/attention_interface.py` + `attention_metadata.py` = the **glue** handing tensors to kernels.
3. **Kernels** — `kernels/` (`mla/`, `ragged_paged_attention/`, `flash_attention/`, fork's `sparse_attn/`): hand-written TPU compute (Pallas/Mosaic — low-level JAX kernel language). Where real optimization lives. Can't run on CPU (MLA kernel has no `interpret` mode) → why we write pure-jnp references.
4. **Runner** — `runner/` (`tpu_runner.py`, `kv_cache_manager.py`): the serving loop — splits prefill (whole prompt at once) vs decode (one token at a time), builds attention metadata (which token → which slot in the paged KV cache), drives the forward.
- **Registry** — `models/common/model_loader.py` maps HF `architectures` string → our model class. Adding GLM = one entry.

So "implement GLM 5.2" = **write `models/jax/glm4_moe_lite.py`** (reuse deepseek modules, make config-driven) + **add the DSA indexer module** + **a registry entry** + **a correctness test**. Kernels mostly exist (MLA) or get ported from the fork (sparse_attn).

## 4. The optimizations, listed
1. **MLA** — KV-cache compression (§2a). *Memory.*
2. **DSA** — sparse attention via the indexer (§2b). *Long-context compute.*
3. **MoE** — top-4-of-64 experts (§2c). *FFN compute.*
4. **Paged KV cache + ragged paged attention** — KV stored in fixed-size "pages" (like OS virtual memory) so many variable-length sequences pack efficiently; one "ragged" kernel handles mixed prefill+decode in a single launch. *Throughput.*
5. **Pallas/Mosaic kernels** — fused, tiled, hand-tuned TPU code; an attention step = one kernel, not dozens of ops. *Raw speed.*
6. **Quantization (fp8/fp4)** — 8/4-bit weights/activations. (Fork's bottleneck was ironically *un-packing* fp4→bf16.) *Memory + bandwidth.* Likely a later phase.
7. **Sharding (tensor + expert parallelism)** — split across TPU chips: heads one way, the 64 experts another (expert parallelism). *Fit + speed.*
8. **Fused projections / expert stacking** — combine small matrices into few big ones (`gate_proj`+`up_proj` → one matmul; all 64 experts stacked into one `[64,…]` tensor). *Fewer, larger, faster matmuls.*

For our **correctness-first** goal, only **1–3 affect the math**; 4–8 are speed/memory, validated separately on TPU.

## 5. What "skipping DSA" actually looks like

"Dense-first, DSA-second" means **Phase 1 builds a complete, runnable GLM that attends to all tokens** — exactly like DeepSeek-V3 does today. In code: the attention module computes Q/K/V via MLA and calls the **normal** attention op over **all** past tokens. **No indexer, no `top_k`, no selection** is wired in — that path just doesn't exist yet. The model is still a correct, complete LM doing full O(n²) attention. **Phase 2** adds the indexer + selection *in front of* the attention, narrowing "all tokens" → "top-2048."

**The key insight that makes this safe:**
> **For any sequence of length ≤ 2048, DSA and dense attention produce the *identical* result** — top-2048-of-(≤2048) selects *everything*. Sparsity only changes behavior once context exceeds 2048 tokens.

Consequences:
1. **Skipping DSA loses nothing mathematically for short contexts.** Phase 1's dense model *is* correct GLM 5.2 for short inputs.
2. **You can validate the DSA machinery on CPU with short sequences:** when seq_len ≤ 2048, assert the sparse path's output == the dense path's output (and the indexer's top-k contains all tokens). Then, with seq_len > 2048, separately assert it selects the *right* tokens. A clean validation ladder instead of "turn on everything and hope."

So "skipping DSA" isn't a downgrade — it's slicing the problem to debug **attention/MoE math** (hard, common bugs) separately from **sparse-selection logic** (different, rarer bugs).

## 6. How predictive is "works on CPU" for "works on the full model on TPU"?

**CPU correctness is a strong, necessary gate for the *model* being right — but not sufficient for the full TPU system.**

**What CPU + HuggingFace-eager comparison validates (the big, common bug class):**
- Architecture assembled correctly (norms, RoPE, head split, scaling, causal mask).
- **Weights mapped correctly** from HF (no transposed matrix, no q-into-k, expert stacking right).
- **Router logic** (sigmoid vs softmax, top-4, correction bias, 1.8 scaling).
- **MoE and MLA math** produce the right numbers.
- (Phase 2) **Indexer math + top-k selection.**

These are the dominant failure modes for a new model — the "outputs fluent garbage" bugs. Matching HF-eager logits on CPU kills ~most of them — call it **70–80% of the "does it work" risk** (intuition, not a measurement).

**What CPU does NOT catch (the TPU-specific risk you chase on hardware):**
1. **Kernel bugs** — CPU runs pure-jnp references; TPU runs Pallas/Mosaic kernels (different code). Mitigated by kernel-vs-reference unit tests.
2. **Sharding / multi-device bugs** — collectives + partitioning; CPU single-device never exercises this (e.g. the fork's multi-batch corruption is invisible on CPU).
3. **Paged-KV / serving bugs** — block-table indexing, prefill↔decode boundary, cache management.
4. **Precision drift** — CPU fp32 vs TPU bf16 (and fp8/fp4). Usually fine, but bf16 can flip a router tie or shift the indexer's top-k.
5. **Performance** — CPU says nothing (the fork was correct *and* 0.43 tok/s).

**Why CPU-first is still right:** laptop iteration is seconds and free; it **separates concerns** (after CPU-correct, any TPU failure is *definitely* kernel/sharding/cache/precision, not "is my router even right"); and **the pure-jnp references you write become the oracles** for validating TPU kernels — reused, not thrown away.

Mental model: CPU = "the recipe is right" (ingredients, steps, proportions); TPU = "cooking it in the real kitchen at scale." Right recipe first ⇒ kitchen problems are *only* kitchen problems.

## 7. What "Lite" means

Confusing naming. In the GLM4-MoE codebase there are **two different architectures**; "Lite" picks one:
- **Plain `Glm4MoeForCausalLM`** (`glm4_moe.py`) = GLM-4.5/4.6/4.7. **Standard multi-head attention** (full Q/K/V, no latent compression, no sparse indexer). The "heavy attention" line.
- **`Glm4MoeLiteForCausalLM`** (`glm4_moe_lite.py`) = attention is **MLA + DSA** (delegates to `DeepseekV2MLAAttention` + `Indexer`). The "light attention" line. **This is GLM 5.2.**

So **"Lite" = the *attention* is lightweight** (MLA shrinks the KV cache ~18×, DSA makes it sparse) — **not** that the model is small/weaker. The transformers default config for the Lite class happens to be a ~30B "GLM-4.7-Flash" (just library placeholder defaults); the **real GLM 5.2 checkpoint** plugs its own (larger) hyperparameters into the **same Lite code path**. We target Lite because that's the architecture with MLA + DSA. The ~30B defaults are useful only as the *smallest ready-made config* to shrink further for CPU tests. (Caveat: vLLM has no literal "GLM-5.2" string — it tops out at GLM-4.7; GLM 5.2 reuses these Lite classes with exact numbers from its checkpoint `config.json`. A new arch string just maps onto this same code path.)
