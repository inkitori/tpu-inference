## 0. Strategy (TPU-first) — read this

Develop and validate **on the real v6e from the start**, running the **real Pallas/Mosaic kernels** — not CPU stand-ins. It is driven by the prior **DeepSeek-V4** bring-up (`/home/enyouki/claude-deepseek-v4`), which was dominated by **multi-device, real-reshard-path bugs CPU structurally cannot reproduce** — chiefly **uninitialized-HBM-on-reshard ("S1")**: a device-side weight reshard read uninitialized HBM and baked per-process garbage into the weights, producing **coherent-looking-but-wrong** output at temp=0 (~30 sessions, root-cause hypothesis overturned repeatedly). This class does **not** appear single-device, and much of it doesn't appear under the CPU `interpret=True` path. So "develop on TPU" means **exercise the real kernels AND the real multi-device sharded weight-loader early** — not "run tiny tests on one chip."

**Three load-bearing guardrails the plan must keep:**
1. A pure-`jnp` fp32 reference **runnable on TPU** — the irreplaceable middle oracle. It runs on TPU not to improve math validation (math is device-independent) but because it is the **kernel's answer key**: the kernel only runs on TPU, so the reference must co-locate with it, else a CPU-vs-TPU fp32 reduction-order delta pollutes the one comparison meant to be clean.
2. **Staged triangulation** (§7) with a tight fp32 *math* gate distinct from a loose bf16 *kernel-noise* gate. Never collapse "kernel-on-TPU vs HF-eager" into one number.
3. **Multi-device** on the real v6e is a **required** gate (§7), not optional — and it must be **armed on a sharding geometry that actually triggers the S1 class** (§6 Phase 1b), not just any multi-chip run. (S1 is a *sharding/reshard* phenomenon, not a host-count one — a single-host **v6e-8** sharded mesh reproduces it; see §1 and `../research/2026-06-20-s1-single-host-reproducibility.md`.)

## 1. Goal & non-goals

### Goal
Bring up **GLM 5.2** — `GlmMoeDsaForCausalLM` (`model_type="glm_moe_dsa"`), i.e. **DeepSeek-V3.2-style MLA + DSA at GLM-5 scale** — as a **native-JAX** model in `tpu-inference`, **developed and validated on TPU with the real kernels from the start. DSA is implemented and validated, not deferred.**

The v1 finish line is **single-sequence correctness of the components on a scaled-down config, on TPU, single- and multi-device** — the dense MLA backbone, the MoE/router stack, and the DSA sparse path — validated against the in-tree `transformers GlmMoeDsaForCausalLM` eager forward via §7. **We do not run the full 78-layer model end-to-end in v1** (that and real-checkpoint generation are Phase 3+).

**Severable v1 fallback (schedule insurance):** if the Mosaic sparse kernel slips, v1 may ship as the **dense backbone (Phase 1) green single+multi-device, with DSA validated in jnp-ref form only** — indexer math + the §2 dense≡sparse equivalence + index-set parity — and the sparse *kernel* dropped to v1.1. The §2 equivalence is exactly what makes a kernel-free DSA milestone meaningful.

### Hardware & environment reality (prerequisite — see Phase 0)
- **Decision: develop v1 on a single-host `v6e-8`** (one process, 8 Trillium chips). The intended ladder: single-device math (Phase 1a) → single-host **8-chip** sharded mesh as the S1 gate (Phase 1b) → multi-host deferred to Phase 3. **Why a single host is enough for S1:** S1 (uninit-HBM-on-reshard) is triggered by the *intra-process device reshard into a sharded layout*, **not** by host/process count — the V4 fork isolated it with a replicated-vs-sharded experiment on a *fixed* mesh, and only ever ran multi-host because its pod couldn't boot smaller (evidence + caveats: `../research/2026-06-20-s1-single-host-reproducibility.md`). 8 chips (vs 4) gives the richer attn_dp+expert sharding geometry that *arms* the gate (§6 Phase 1b). (Box/runtime specifics — which v6e-8, how to bring it up — live in `CLAUDE.md`, not here.)
- **Env:** system Python is 3.10 and **cannot** install the pinned `jax==0.10.1` (needs ≥3.11; repo Docker uses 3.12); a py3.12 venv already exists and is reused in Phase 0. The persistent XLA **compilation cache is on by default** (`compilation_manager.py`), so edit→run is near-instant after the first cold compile.

### Non-goals (deferred past v1)
- **Full end-to-end model** (all 78 layers, real-checkpoint generation). Phase 3.
- **MTP** — the checkpoint's `model.layers.78` MTP head is **dropped on load** (`_keys_to_ignore_on_load_unexpected = [r"model\.layers\.78.*"]`).
- **Quantization** (fp8 UE8M0 / fp4). v1 runs bf16/fp32; the GPU DSA path is fp8 (DeepGEMM `fp8_mqa_logits`), v1 uses a bf16/fp32 indexer.
- **Aggressive kernel autotuning** — kernels are idiomatic-from-the-start; autotuning is bounded and post-correctness (§10).
- **Multi-batch / concurrent decode — a v1 SCOPING decision, not the production target.** The repo serving path (`runner/persistent_batch_manager.py`, `input_batch.py`, `decode_loop.py`) is multi-request and production needs batching (see the new §11 production roadmap). v1 is single-sequence only because the fork has a **confirmed** `max_num_seqs>1` MoE-corruption bug, which is the **same uninit-HBM/reshard mechanism as S1**. The Phase-1b S1 work (real sharded loader, host-stack-into-sharded-layout, N-dev==1-dev fp32 gate; §6/§8) is exactly the prerequisite that retires this corruption — so concurrent decode is a **bounded follow-on (§11 production phases), not a permanent non-goal nor a separate hard problem**, once Phase 1b is green.

## 2. Architecture summary

GLM 5.2 is a decoder-only transformer where every layer is **MLA + DSA**, the FFN is **MoE** (after a few leading dense layers):

1. **MLA (Multi-head Latent Attention)** — cache one small **latent** (`kv_lora_rank=512` + shared `qk_rope_head_dim=64` rope key = 576 numbers) and **up-project** (`kv_b_proj`) to per-head K/V at attention time. Query path is compressed (`q_a_proj → q_lora_rank → q_a_layernorm → q_b_proj`). RoPE (**interleaved** — pairs adjacent dims, `apply_rotary_pos_emb_interleave`) applies only to the 64-dim rope slice for MLA; the indexer uses **rotate-half** (opposite convention, §2 item 2 (DSA) / §4 item 5 (RoPE)). The attention splits **nope-first** `[qk_nope=192, qk_rope=64]`. Softmax scale = `qk_head_dim**-0.5 = 256**-0.5` (**no mscale** under default rope; §4 item 9 (softmax scales)). **No per-head QK-norm** — only `q_a_layernorm` and `kv_a_layernorm`.
2. **DSA (lightning indexer)** — a cheap auxiliary scorer ranks past tokens and selects the **top-`index_topk`=2048**; MLA then runs only over the selected set (additive `-inf` mask combined with the causal mask). V3.2/GLM form: per-head dot-product, **`relu`** on the scores, weighted head-sum, **no softmax**; `k_norm`=**LayerNorm** (the one LayerNorm in an otherwise RMSNorm model). The indexer splits **rope-first** `[qk_rope_head_dim, index_head_dim - qk_rope_head_dim]` = `[64, 64]` here (opposite order to main attention) — the second chunk is `index_head_dim - qk_rope_head_dim` (the non-rope "pass" part), **not** a second rope slice, so any tiny/test config must keep `index_head_dim >= qk_rope_head_dim` (§6 keeps `index_head_dim=128`, satisfying this; HF: `q_rot,q_pass = split(q,[qk_rope_head_dim, head_dim - qk_rope_head_dim])`, `modeling_glm_moe_dsa.py:233,:236`) — and **reuses the MLA query LoRA residual** `q_a_layernorm(q_a_proj(x))`. Some layers (`indexer_types[i]=="shared"`) **reuse the previous layer's top-k**.
3. **MoE** — 256 expert FFNs (`moe_intermediate_size=2048`) + 1 always-on shared expert; a router picks **top-8-of-256**. **sigmoid** scoring; `e_score_correction_bias` added **only for selection** (gathered weights use bias-free sigmoid scores); `norm_topk_prob` renormalizes; routed output scaled by `routed_scaling_factor` (default 2.5); the shared expert added **unscaled**. The first `first_k_dense_replace` (default 3) layers are a plain dense SwiGLU FFN.

**Key safety property (drives the validation ladder):** for any sequence of length ≤ `index_topk` (2048), `topk = min(index_topk, total_len)` selects **every** valid token, so DSA and dense attention produce the **identical** result. Sparsity only changes behavior past 2048 tokens. This lets us validate a correct **dense backbone** first (short seqs), then the **sparse path** (long seqs), with a clean equivalence check at the boundary — checkable **on TPU at fp32** (≈1e-6), isolating the entire sparse-plumbing bug class with no HF dependency.

## 3. Exact GLM 5.2 config (`GlmMoeDsaConfig`)

`GlmMoeDsaConfig` subclasses `Glm4MoeLiteConfig`; the generated config flattens inherited fields. **Defaults below verified live** via `GlmMoeDsaConfig()`. The real checkpoint's `config.json` is authoritative and overrides these; our config path reads everything from `hf_config`. DSA is gated by **`is_v32 = hasattr(config,"index_topk")`** — the `Glm4MoeLiteConfig` base has **no indexer**.

| Field | Default | Field | Default |
|---|---|---|---|
| `hidden_size` | 6144 | `n_routed_experts` | 256 |
| `num_hidden_layers` | 78 | `n_shared_experts` | 1 |
| `intermediate_size` (dense FFN) | 12288 | `num_experts_per_tok` | **8** |
| `moe_intermediate_size` | 2048 | `n_group` / `topk_group` | **1 / 1 (grouping OFF)** |
| `vocab_size` | 154880 | `routed_scaling_factor` | **2.5** |
| `max_position_embeddings` | 202752 | `norm_topk_prob` | True |
| `num_attention_heads` | 64 | scoring | **sigmoid (hardcoded; `scoring_func` is a dead key — see below)** |
| `num_key_value_heads` | 64 | `rms_norm_eps` | 1e-5 |
| `q_lora_rank` | 2048 | `attention_bias` | False |
| `kv_lora_rank` | 512 | `tie_word_embeddings` | False |
| `qk_nope_head_dim` | 192 | `qk_head_dim` (derived) | 256 |
| `qk_rope_head_dim` | 64 (`head_dim` attr → this) | `initializer_range` | 0.02 |
| `v_head_dim` | 256 | `hidden_act` | "silu" |

**RoPE config key is `rope_parameters`** (a dict), **not `rope_scaling`**. Default `{'rope_theta': 10000.0, 'rope_type': 'default'}`. ✅ **The real `zai-org/GLM-5.2` `config.json` uses `{'rope_theta': 8000000, 'rope_type': 'default'}` — plain RoPE, NO YaRN** (verified). So the YaRN/`mscale²` path is **never taken** for this checkpoint (§4 item 9 (softmax scales), §9 — resolved). Other real-checkpoint overrides vs the §3 defaults: `max_position_embeddings=1048576` (not 202752), and `head_dim=192` (=`qk_nope_head_dim`, not the 64 default) — **don't rely on `config.head_dim`; size MLA from `qk_head_dim=256`/`qk_nope=192`/`qk_rope=64` explicitly**.

**Layer schedules** (two independent per-layer patterns, computed in `__post_init__`):
- **Dense-FFN vs MoE → `first_k_dense_replace` (default 3) → `mlp_layer_types`**: `first_k_dense_replace` **IS** a first-class HF config field (`configuration_glm_moe_dsa.py:128`) and the **primary driver** — `__post_init__` derives `mlp_layer_types = ["dense"]*min(first_k_dense_replace,L) + ["sparse"]*rest` (`:151-154`) ⇒ **first 3 dense**, rest MoE. Both keys always coexist; **read `first_k_dense_replace` directly** (or accept an explicit `mlp_layer_types` override if the checkpoint supplies one). vLLM keys on the same `first_k_dense_replace`.
- **Indexer recompute vs reuse → `indexer_types`**: computed in `__post_init__` (`configuration_glm_moe_dsa.py:135-146`) from `index_topk_pattern` (explicit `'F'`/`'S'` string) if present, else from `index_topk_freq` (default 1) **and** `index_skip_topk_offset` (default 2) via `'full' if (max(i-offset+1,0) % freq)==0 else 'shared'`. ⚠️ **With the default `index_topk_freq=1`, every layer is `"full"`** (no `"shared"`) — so the scaled-down test config **must** set `index_topk_freq>1` (or an explicit `index_topk_pattern`) to produce ≥1 `"shared"` layer and exercise the reuse path. A `"shared"` layer reuses the previous layer's `topk_indices` (`skip_topk`). **Every attention layer is DSA-sparse**; "full"/"shared" only controls whether the *indexer scores* are recomputed. **Read from `hf_config`**. ✅ **Real GLM-5.2:** `index_topk_freq=4`, `index_skip_topk_offset=3` → `indexer_types` = `"full"` at layers 0,1,2 then every 4th (6,10,14,…), the rest `"shared"` (the checkpoint ships the explicit 78-entry array). The real model is **mostly `"shared"`** — the reuse path is heavily exercised; the tiny test config should set `index_topk_freq=4`/`index_skip_topk_offset=3` to mirror it.

**DSA indexer fields:** `index_topk=2048`, `index_head_dim=128`, `index_n_heads=32`, `index_topk_freq=4`, `index_skip_topk_offset=3`, `indexer_types` (explicit 78-entry array in the real config). All read from `hf_config`; never hardcoded.

**Dead config keys (present on the real checkpoint but the HF `glm_moe_dsa` code never reads them):** `rope_interleave=true`, `indexer_rope_interleave=true`, `pretraining_tp=1`, `scoring_func="sigmoid"`. They are **absent** from a default `GlmMoeDsaConfig()` (stored only as unknown kwargs when the checkpoint sets them) — so the old "removed → `AttributeError`" framing is **wrong**: on the real config they are live attributes. The HF forward **hardcodes** its RoPE conventions regardless of these flags. (`scoring_func` is the same story: the real `config.json` sets it to `sigmoid` (line 215), but `route_tokens_to_experts` hardcodes `router_logits.sigmoid()` (`modeling_glm_moe_dsa.py:591`) with no branch on it — sigmoid scoring is correct regardless.) ⚠️ `indexer_rope_interleave=true` is **load-bearing for vLLM, not HF** — vLLM honors it (indexer → interleaved) while HF ignores it (indexer → rotate-half): a real HF-vs-vLLM divergence on the real checkpoint (§4 item 5 (RoPE), §9).

**Norms — definitive** (one wrong eps = silent math-gate failure; this is the canonical mapping — §7/§9 reference it):

| param name | class | dim | eps | weight | bias | source |
|---|---|---|---|---|---|---|
| `self_attn.q_a_layernorm` | RMSNorm | `q_lora_rank=2048` | **1e-6** (class default, NOT `rms_norm_eps`) | yes | no | `modeling_glm_moe_dsa.py:377,:49` |
| `self_attn.kv_a_layernorm` | RMSNorm | `kv_lora_rank=512` | **1e-6** (class default) | yes | no | `:385,:49` |
| `self_attn.indexer.k_norm` | **LayerNorm** | `index_head_dim=128` | **1e-6** | yes | **yes** | `:193` |
| `input_layernorm`, `post_attention_layernorm` | RMSNorm | `hidden_size=6144` | `rms_norm_eps`=**1e-5** | yes | no | `:637,:638` |
| `model.norm` (final) | RMSNorm | `hidden_size=6144` | `rms_norm_eps`=**1e-5** | yes | no | `:716` |

⚠️ **HF-vs-vLLM disagree on `q_a_layernorm`/`kv_a_layernorm`:** vLLM uses `rms_norm_eps` (1e-5) for these two; **match HF's 1e-6.** `k_norm` is the **only LayerNorm** in an otherwise all-RMSNorm model (the only norm with a bias). **No per-head q/k norm.** The three `rms_norm_eps`=1e-5 rows are the **only** norms reading `rms_norm_eps`.

## 4. GLM/DSA deltas vs the existing DeepSeek-V3 JAX path (the things to get exactly right)

The repo's `models/jax/deepseek_v3.py` is **plain DeepSeek-V3 (no indexer)**; its module globals are already close to GLM (`routed_scaling_factor=2.5`, `num_experts_per_token=8`, `first_k_dense_replace=3`, 256 experts). Each delta below is a known foot-gun.

1. **Config-driven hparams** — replace DeepSeek's hardcoded module globals (`deepseek_v3.py:76-108`) with values from `hf_config` (à la `qwen3_moe.py`). GLM differs in `n_group` (1 vs 8), `hidden_size` (6144 vs 7168), `q_lora_rank` (2048 vs 1536), `qk_nope_head_dim` (192 vs 128), `v_head_dim` (256 vs 128), `num_attention_heads` (64 vs 128), and adds the `index_*` fields.
2. **`n_group = topk_group = 1`** — grouping **off**; take top-8 from all 256. `DeepSeekV3Router.get_topk_indices` already skips the group branch when `n_groups==1`.
3. **Router gate in fp32**, **sigmoid** scoring, `e_score_correction_bias` added **before top-k for selection only**; **gathered weights use bias-free sigmoid scores**; `norm_topk_prob` renorm (`/ (sum + 1e-20)`).
4. **`routed_scaling_factor` (default 2.5) applied to the routed-expert output**, **not** to router logits; shared expert added **unscaled**. Site already correct in `SharedFusedMoe.__call__`; pass the config value through `DeepseekV2Moe`.
5. **RoPE — MLA is INTERLEAVED, the indexer is rotate-half.** **The single highest porting risk.** MLA uses `apply_rotary_pos_emb_interleave` (pairs adjacent even/odd dims; `modeling_glm_moe_dsa.py:434`, fn `:302-338`); the indexer uses rotate-half `apply_rotary_pos_emb` (`:239`, HF's own comment at `:238` confirms it). ⚠️ **MLA: HF and vLLM AGREE** (both interleaved; vLLM MLA `is_neox_style=False` hardcoded). **Indexer: they DIVERGE on the real checkpoint** — HF `glm_moe_dsa` **hardcodes** the indexer to rotate-half (`:239`) and **ignores** `config.indexer_rope_interleave`, but vLLM honors it (`is_neox_style = not indexer_rope_interleave`, `deepseek_v2.py:1028`) and GLM-5.2's `config.json` sets `indexer_rope_interleave=true` → vLLM runs the indexer **interleaved**. **v1 follows the HF oracle (rotate-half indexer)**; reconciling against the real weights is the #1 Phase-3 item (§9). For MLA, port the **interleaved** layout (the repo's `DeepseekScalingRotaryEmbedding` has the correct interleave for MLA, BUT its `_compute_inv_freq`/`_compute_sin_cos` apply YaRN unconditionally — interpolation blend, `mscale`, `t=arange(orig_max*scaling_factor)`. For v1 default-rope, construct it with `scaling_factor=1` (which collapses the YaRN math to plain RoPE: interpolation==extrapolation, mscale=1, t=arange(orig_max)); `mscale_value` already defaults to 1. It is NOT a drop-in plain-RoPE class — `scaling_factor` is a required ctor field. Alternatively port `apply_rotary_pos_emb_interleave` standalone.); use the base rotate-half `RotaryEmbedding` only for the **indexer**. Pin **two** bit-for-bit tests: MLA vs `apply_rotary_pos_emb_interleave`, indexer vs `apply_rotary_pos_emb` (§7). Main attention splits **nope-first** `[192,64]`; indexer splits **rope-first** `[qk_rope_head_dim, index_head_dim - qk_rope_head_dim]` = `[64, 64]` (the second chunk is the non-rope "pass" part, not a second rope slice — see §2 item 2 (DSA)).
6. **silu only** for FFN/expert activation (gated SwiGLU).
7. **Dense-then-MoE schedule** — `first_k_dense_replace` **IS** an HF config field (default 3; `configuration_glm_moe_dsa.py:128`) and the **primary driver**: HF derives `mlp_layer_types` from it in `__post_init__` (`:151-154`). Read `first_k_dense_replace` directly; accept an explicit `mlp_layer_types` override if the checkpoint supplies one (§3). vLLM keys on the same field.
8. **No per-head QK-norm** (§3).
9. **Two distinct softmax scales — don't conflate them.** The **MLA attention** scale = `qk_head_dim**-0.5 = 256**-0.5` (`modeling_glm_moe_dsa.py:398`); the **DSA indexer** scale is separate = `index_head_dim**-0.5 = 128**-0.5 ≈ 0.0884` (`:195`), applied fp32 at `:246`. Under **YaRN**, HF multiplies the **MLA** scale by `mscale²` (`:398-404`) — and **vLLM does the identical thing** (`deepseek_v2.py`), so HF and vLLM **AGREE** (no longer an open divergence). The mscale branch touches **only** the MLA scale; the **indexer scale is a fixed `128**-0.5`, no mscale ever**. Under default rope both leave the bare MLA scale. v1 tests use default rope. ✅ **Resolved:** the real `zai-org/GLM-5.2` `config.json` uses `rope_type="default"` (`rope_theta=8M`) — YaRN is **not** enabled, so the mscale branch is **never taken** for this checkpoint (§9).
10. **DSA indexer = V3.2/GLM form, NOT the fork's V4 form.** Keep: q-up-proj from the shared q-LoRA residual, **`relu` on scores**, per-head `weights_proj × n_heads**-0.5`, head-sum, `top_k`. **Drop** (V4-only — these live in the fork's compressor/indexer path and are simply *not* reimplemented, since the GLM indexer is built fresh from the HF oracle, not ported line-by-line): the **compressor/CSA** (compressor-side attention over compressed KV — GLM's K is instead direct `k_norm(wk(hidden_states))` per-token, the single biggest math change), **SWA** (sliding-window-attention variants), **mHC sinkhorn** (multi-head-combine Sinkhorn normalization, fork `hc_split_sinkhorn`), and **APE** (additive per-window positional bias, fork `CompressorParams.ape`). `k_norm` is **LayerNorm**, not RMSNorm. **fp32-pinned indexer (production bf16 path):** HF pins **`indexer.weights_proj` to fp32** (`_keep_in_fp32_modules`, `modeling_glm_moe_dsa.py:692`) and runs the **entire indexer score + weight/head-sum reduction in fp32** regardless of model dtype — `scores=relu(q.float()@k.float())*scale` (`:246-247`), `weights=weights_proj(x).float()*n_heads**-0.5` (`:250`), `index_scores=weights@scores` (`:251`). The fp32 jnp-ref oracle gets this for free, but the **production bf16 indexer MUST mirror it** (compute weights_proj, scoring, and head-sum reduction in fp32, downcast only after top-k) to hit the §7 1e-3 fp32 gate. `e_score_correction_bias` is fp32-strict (`_keep_in_fp32_modules_strict`, `:690`; consistent with §4 item 3 router-gate-in-fp32). **Indexer causality (pre-topk, distinct from the MLA mask):** the indexer must add a causal additive mask to its OWN fp32 `index_scores` BEFORE `top_k`, or the selected set can include future tokens (breaking the §2 dense==sparse equivalence). HF derives it as the head-0 slice of the 4D causal mask — `indexer_mask = attention_mask[:, 0, :, :]` (`modeling_glm_moe_dsa.py:445`), added at `:254-255`; when no mask is supplied it builds causality from positions via strict inequality `key_positions[None,None,:] > position_ids[:,:,None]` + `masked_fill(..., -inf)` (`:257-259`), then `topk` runs over the masked scores (`:261-262`). This is separate from the additive `-inf` MLA mask built from the selected indices (the attention path rebuilds its own causal index_mask at `:467-468`). The jnp indexer ref must replicate the strict `>` causal cutoff on fp32 scores pre-topk.
11. **PyTorch quirks — nothing to port:** do **not** carry over DeepSeek-V3's layer-0 residual `.clone()` or fp16-overflow rescale; the HF GLM-DSA port already dropped them (verified — `DecoderLayer.forward` is a plain residual add).

## 7. Testing & validation strategy

**Oracle = the in-tree `transformers GlmMoeDsaForCausalLM` eager forward** (torch, CPU, random weights, no download) — an **independent** implementation of every §4 delta. It runs on the host; the model-under-test runs on TPU; comparison is a **host-side `maxabs`** (JAX arrays → host numpy). Upcast both sides to fp32 **inside** `maxabs` (a bf16 TPU output returns as numpy bf16 — load-bearing). No `jax_enable_x64`.

### The triangulation (the core of the strategy)
Three implementations, three single-delta comparisons — never a fused HF-vs-bf16-kernel verdict:

```
HF-eager (CPU, fp32)  ──[MATH gate, 1e-3]──►  jnp-ref (TPU, fp32)  ──[KERNEL gate]──►  Pallas kernel (TPU)
        independent oracle                      our answer key                          shipped path
```

- **MATH gate** (HF-fp32 vs jnp-ref-fp32-on-TPU, rtol≈1e-3): catches every §4 math/spec bug (RoPE convention, head splits, `256**-0.5` scale, kv_b split, router bias-for-selection, routed_scaling 2.5, relu placement). This is the gate with *teeth* — prove it by an **injected-1%-error test** (perturb a scale, assert the fp32 gate catches it). Without an fp32 reference on TPU there is **no** math gate: the bf16 kernel noise floor (~4e-3/element, compounding over depth) is larger than a real ~1% bug.
- **KERNEL gate** (jnp-ref vs kernel, on TPU): run the kernel **twice** — (a) fp32 vs jnp-ref-fp32 at tight tolerance to check kernel *algebra* — **cast the `q`/`ql_nope` inputs to fp32** (the kernel's fp32 output path is gated on `q_dtype`, `kernel.py:1938`; `:370` asserts output dtype == q_dtype), in addition to `s_dtype=fp32, p_same_dtype_as_v=False`; (b) bf16 shipped defaults vs jnp-ref-bf16 at the **empirically-measured bf16 floor** (run the jnp-ref in bf16 to derive the floor — don't guess). Repo precedent: bf16 attention ~0.1–0.2 (`ragged_paged_attention_kernel_v2_test.py:129-134`, `mla_v2_test.py:405`); MoE/elementwise ~5e-2.
  - ⚠️ The kernel gate carries **double duty**: the non-absorbed jnp-ref deliberately does NOT absorb, so a bug in the **kv_b absorption** algebra shows up only here and the jnp-ref can't catch it. Give the **fp32 kernel-algebra check the same teeth** as the math gate: a **second injected-error test that perturbs the kv_b absorption split** and asserts the fp32 kernel gate catches it.

### Tiered comparison (each submodule at the layer where it lives, plus full forward)
| Check | Tolerance / method |
|---|---|
| RoPE — MLA interleaved, indexer rotate-half (two tests) | exact / 1e-6 |
| Router | top-k indices **exact** (fp32 gate, well-separated scores) + weights 1e-2 |
| RMSNorm / LayerNorm, dense FFN, embed/lm_head (per-norm eps/class/bias → the §3 "Norms — definitive" table) | 1e-3 (fp32 math gate) |
| MLA attention | math gate 1e-3 (jnp-ref vs HF); kernel-algebra 1e-3 fp32 (+ absorption injected-error test); shipped bf16 at measured floor |
| MoE experts (vs **eager** `GlmMoeDsaNaiveMoe` — assert `config._experts_implementation == "eager"`) | 1e-2 |
| Indexer (`"full"` layers; absent on `"shared"`) | **boundary-aware tie-tolerant** top-k set equality + index_scores 1e-3 fp32 (the gate covers the **pre-topk causal mask** — the masked fp32 `index_scores`, so a missing causality cutoff trips it) |
| Dense==sparse (§2, seq ≤ topk) | **fp32 ≈1e-6 on TPU** — isolates sparse plumbing, no HF dependency |
| Full forward | per-layer index-**set** equality (tie-tolerant) + fp32 math gate 1e-3 / bf16 shipped at floor + argmax ≥0.95 |

### Multi-device (required gate, not optional)
The bug class that motivated TPU-first lives here. S1 is a *sharding/reshard* phenomenon, not a host-count one — a **single-host `v6e-8` (8-chip) sharded mesh reproduces it** (evidence: the V4 fork's replicated→no-S1 vs sharded→S1 experiment on a fixed mesh; `../research/2026-06-20-s1-single-host-reproducibility.md`). On the **8-chip mesh**, via the **real sharded weight-loader**:
- **N-device fp32 result == 1-device fp32 result** (sharding must be value-invariant). Compare 1/2/4/8-device (the full 8-chip mesh arms the richest sharding geometry per §1).
- **NaN-poison HBM** before weight load; assert no NaN/garbage leaks into outputs (catches NaN-surfacing variants of S1; the per-weight 1-dev-vs-N-dev checksum below catches the coherent-but-wrong variant; the fix is host-stacking into the sharded layout, not device reshard — §5, where `gmm_v2(zero_initialize=True)` is noted as disproven).
- **When the gate trips, localize it (don't bisect by deduction).** Pair the pass/fail gates with localizing probes — the V4 fork's S1 took ~30 sessions in deductive-only reasoning and was cracked in one session only once a per-weight/per-stage checksum probe was built (`[ckSPLIT]`, fork commit `c98430d8`): (1) **per-parameter post-load checksum, 1-dev vs N-dev** — a read-only `shard_map` returning reorder-immune `[gsum, sqsum, absmax]` per loaded weight (esp. stacked routed expert weights); the first weight whose sqsum/absmax differs pins the corrupted weight + sharding axis (this isolated S1 to routed W1/W3 axis-1 leaves while W2 axis-0 was byte-identical); (2) **per-stage / per-layer activation-divergence sweep** — checksum each einsum stage 1-dev vs N-dev to find the first diverging stage, distinguishing a corrupted weight load from a corrupted compute/collective; (3) **dump the realized sharded layout** (`jax.debug.visualize_array_sharding` / per-shard host gather) for the pinned weight. ⚠️ **NaN-poison is necessary but not sufficient:** the real S1 corruption was *coherent finite garbage* (absmax flips, ~20% gsum drift), not NaN — the value-localizing checksum, not NaN-poison, catches the coherent-but-wrong class.
- Run the gate against the **sharding-geometry stress fixture** (§6 Phase 1b), not just the tiny config, or it may pass vacuously. Exercise `expert_axis_name` partitioning. **A single-device v6e pass is NOT TPU validation.**

### Determinism & ties
TPU is seeded-reproducible but **not bit-equal** to the torch ref; reductions are non-associative and order differs (more so multi-device). Top-k ties are a documented hazard (`fused_moe_v1_test.py:299-300`, jax #34620). Hence: the fp32 math gate uses well-separated scores where exact equality is asserted; the indexer set compare is **boundary-aware tie-tolerant** (adjudicate boundary swaps by recomputed fp32 scores); `index_scores` gated separately at 1e-3 so a relu/scale bug shows up even when the set survives. Pin the torch ref: fixed seeds, single-process (`WORLD_SIZE=1`), fp32, eager.

### CI / test placement
All CI runs on **TPU agents** (`.buildkite/pipeline_jax.yml`) — there is **no** CPU/non-TPU runner, so no skip-if-no-TPU marker is needed.
- **Single-device tests** (math gate, weight-map golden, construction regression) go in `tests/models/jax/test_glm_moe_dsa.py`; they are **auto-collected** by the existing step `${TPU_VERSION}_test_7_2` (`pytest .../tests/`, queue `${TPU_QUEUE_SINGLE:-tpu_v6e_queue}`, single chip) with **no pipeline edit**. Mind the `--fail-under=68` coverage gate in step `test_7_3`. If a new test is broken-by-import (as `test_deepseek_v3.py` is), add it to the `--ignore=` list.
- **Multi-device v6e-8 gate** (Phase 1b/§7, S1/reshard): add a **dedicated step** mirroring the existing collective-kernels step `${TPU_VERSION}_test_9` — queue `${TPU_QUEUE_MULTI:-tpu_v6e_8_queue}`, gated via `if: build.env("NIGHTLY") == "1" || "${RUN_GLM_DSA_MULTIDEVICE:-0}" == "1"` so it does **not** provision the 8-chip queue on every PR — pointed at the multi-device GLM/DSA test (N-dev==1-dev fp32 + NaN-poison + sharding-geometry stress fixture).

## 10. Kernel optimization — idiomatic now, bounded tuning later

Performance is pursued in two modes; neither is a continuous autonomous loop during correctness bring-up.

**Now (free, during bring-up):**
- Write kernels in the **performant TPU idiom from day 1** (sensible block/tile structure, avoid pathological layouts, `shard_map`-wrapped Mosaic, host-side RoPE-freq precompute).
- **Tuned-param reuse is a no-op for GLM v1 — do not rely on it.** Every entry in `tuned_params.py` is keyed on `actual_num_q_heads=128` and fp8 `q_dtype`/`kv_dtype` (DeepSeek-V3 decode shapes). GLM has `num_attention_heads=64` and v1 runs **bf16** (§1 non-goals), so the `TuningKey` (which hashes head count AND dtype) never matches and `get_tuned_params` (`:379`) silently falls to hardcoded defaults (`num_kv_pages_per_block=3, num_queries_per_block=1, decode_batch_size=4`). v1 therefore runs the **un-tuned kernel defaults**.
- **One-time structural perf smoke for the DSA sparse kernel** across a **short sweep of context length N (the kernel's true cost axis — the one-hot reads all N, not just `index_topk`)** at real `index_topk=2048`: at least N≈2048 (all-N==topk, no cliff) and N at a representative production length (N ≫ topk, where the all-N read dominates), to confirm the one-hot structure is within an order of magnitude of viable **and to size when the DMA scalar-prefetch rewrite (§5) becomes mandatory** — caught while a rewrite is still cheap. (Fixing only topk=2048 would miss the N cliff entirely.) (MLA needs no such smoke — its structure is proven.)

**Later (bounded, post-correctness):** *autotuning* starts only **after single-sequence correctness is green and the kernel spec is frozen** (notably the YaRN `mscale²` question, §9). Then:
- Run **bounded, human-triggered sweeps** at **real GLM-5.2 dims** (tiny-config perf is meaningless — the serving lookup is exact-shape-keyed) using the existing tuner (`tools/kernel/tuner/v1`, `--run_locally=True`). **GLM-shaped tuning is REQUIRED for any perf claim, not optional polish** (the inherited table matches none of GLM's shapes — see "Now"). Add GLM-shaped keys to `tuned_params_mapping` (`actual_num_q_heads=64`, `actual_lkv_dim=512`, `actual_r_dim=64`, bf16 `q_dtype`/`kv_dtype`/`s_dtype`, real `page_size`/`kv_packing`) before the sweep, since the serving lookup is exact-shape-keyed and the current table covers none of GLM's shapes.
- For DSA, **write a dedicated `sparse_attn` tuner** (subclass `KernelTunerBase`) once the kernel is correct — the existing tuner is MLA-decode-only with no feedback loop.
- **Guardrails (mandatory):** a tuned config is accepted only if the **full correctness suite still passes**; a **tuned-vs-baseline regression test** (≤5% slower, modeled on `mla_tuned_vs_baseline_test.py:131-135`, extended to GLM/DSA shapes); tuning runs against a **frozen kernel commit**; apply-back is **human-reviewed**, never silently mutated.
- **Stop criterion:** discrete bounded sweeps — stop when the best config is within timing noise (<3–5%) of the previous best, or at a fixed TPU-hour budget.

## 11. Production roadmap (post-v1) — sharded, batched, optimized

v1 (Phases 0–3) is **single-sequence correctness**, deliberately. Production-readiness — concurrent decode, continuous batching, fp8 — is staged as explicit phases **built on the Phase-1b S1 fix**, not a permanent non-goal (§1 re-scope). Each phase keeps the §0 guardrails (jnp-ref fp32 oracle on TPU; fp32 math gate distinct from bf16 kernel gate; N-dev==1-dev value-invariance) and the single-chip→multi-chip ladder. The new §12 mesh section is the sharding contract these phases consume.

### Phase 4 — concurrent decode (`max_num_seqs>1`) on the v6e-8
**Why it is bounded, not a new hard problem:** the fork's confirmed `max_num_seqs>1` MoE corruption is the **same uninit-HBM/reshard mechanism as S1** (fork commit `fc03c1f9`). Phase-1b retires S1 (host-stack-into-sharded-layout; N-dev==1-dev fp32; per-weight checksum probe, §7). Concurrent decode reuses the repo's existing multi-request serving path — `runner/persistent_batch_manager.py`, `runner/input_batch.py`, `runner/tpu_runner.py::_prepare_inputs` (which already builds `request_distribution=[num_decode,num_decode,num_reqs]`, `seq_lens[max_num_seqs]`, `query_start_loc[max_num_seqs+1]`) and the per-seq-looping MLA kernel (`kernels/mla/v2/kernel.py`, `page_indices_offset = seq_idx*pages_per_seq` `:704`).
**DSA-specific work (the genuinely new part):**
- **`topk_indices` coordinate space under paging — MUST be per-request-local** (see §12). The fork's sparse kernel uses per-batch-slot offsets into a contiguous `kv[B,N,D]` (cross-request isolation is free). In the repo's flat-ragged-paged world there is no per-request contiguous KV — the kernel must translate a selected per-request token position to a physical paged slot via the **block_table**. Decision: `topk_indices` are **per-request-local (0..seq_len-1)**; the kernel demuxes per `seq_idx` using `cu_kv_lens` + `block_table` exactly as the MLA kernel does, so a selected index can never cross a request boundary. (Global-flat-physical indices are rejected: they require a cross-request-boundary guarantee the indexer cannot give.)
- **Indexer-key KV cache as a registered cache group** (see §12): it must participate in block allocation, continuous batching, preemption.
**Exit (4):** `max_num_seqs>1` decode on the v6e-8 — N-dev==1-dev fp32, NaN-poison clean, per-request outputs bit-stable vs single-sequence runs of the same prompts (the §7 value-invariance gate extended to the batch dim).

### Phase 5 — continuous batching + the in-graph decode loop
The repo has a dedicated high-throughput decode fast path: `tpu_runner.py` gates `is_decode_only and self.enable_continue_decode → _execute_continue_decode`, which drives `runner/decode_loop.py::continue_decode` — a `jax.lax.while_loop` whose carry (`TpuSamplingState`) includes the full `AttentionMetadata` pytree, **rebuilt inside the loop body each step**.
**DSA consequence (a constraint v1 never surfaced):** adding `topk_indices` to `AttentionMetadata` (§5) makes it **loop-carried state**, so the §5 runner-side population does NOT work on this path — the runner cannot repopulate the field across an on-device multi-step loop. On `continue_decode`, the indexer scores + top-k must be computed **device-side inside the loop body** each step, from the indexer-KV cache that grows one row per step; the indexer-KV append and the `-1`-sentinel topk buffer must be expressible as loop-carried, fixed-shape state. v1 leaves `enable_continue_decode=False` (default), so v1 correctness is unaffected.
**Exit (5):** DSA correct on `_execute_continue_decode` (jnp-ref value-match at fp32 over a multi-step decode), throughput within the repo's existing continuous-decode regression envelope.

### Phase 6 — fp8 quantization of the DSA path (see §13)
Brings the indexer (and, where validated, MLA) to the GPU reference's fp8 regime. Gated behind Phases 4–5 and its own correctness ladder.

**Ladder discipline (unchanged):** every production phase is single-chip-correct first, then re-gated on the 8-chip mesh (and the §12 stress geometry), then — only at Phase 3/§14 — multi-host. No phase ships on a perf number without its correctness suite green (§10 guardrails).

## 12. Production sharding / parallelism mesh (TP / EP / DP / SP)

The repo's six logical mesh axes (`layers/common/sharding.py`, `MESH_AXIS_NAMES=('data','attn_dp','attn_dp_expert','expert','model','dcp')`) and their `ShardingAxisName` specs are reused **by import**; v1 inherits the DeepSeek wiring unchanged. This section states the **degrees and the tensor→axis contract** for GLM, and fills the gaps the inherited wiring leaves (indexer, topk_indices). Mesh **degrees** are a runtime knob, not a design decision: they come from `vllm_config.parallel_config` + `additional_config['sharding']['sharding_strategy']` (`ShardingStrategy.from_vllm_config`), so the Phase-1b gate sets them per-run with no code change.

### Inherited contract (reused unchanged from `deepseek_v3.py`)
- **MLA attention.** Query `query_nth = P(None, ATTN_DATA, None)`; kv_cache `P(BATCH, None, ATTN_HEAD)`. The kernel runs as **MQA with one kv head** (`num_key_value_heads=1` when `use_mla_kernel`), so the **576-wide latent cache stays REPLICATED across the model/TP axis** — KV memory does **not** shrink with TP; **attention-data-parallel (`attn_dp`) is the KV-memory lever**, exactly the DeepSeek path (`sharding.py` sets `num_kv_heads=1` for MLA precisely so TP does not duplicate the single KV head). The split `k_up`/`v_up` projections shard on `ATTN_HEAD`.
- **64 query heads shard on the model axis** → **64 must be divisible by the model-axis size** on the 8-chip mesh; verify before Phase 1b. (GLM 64 heads vs DeepSeek 128 changes divisibility and attn_dp choices — re-derive, don't copy DeepSeek's.)
- **Routed experts.** `SharedFusedMoe` picks `moe_edf`/`moe_efd` specs off `ATTN_DATA_EXPERT`/`MOE_TENSOR` gated by `use_ep`; `expert_axis_name = ATTN_DATA_EXPERT`.

### Expert sharding — 256 experts, EP vs TP mutual exclusion
- **Production EP is even:** `n_routed_experts=256` divides cleanly across v6e-8 (32 experts/shard) and larger power-of-two slices — **no empty/uneven shards in the real model**.
- **Current-code constraint (verified `deepseek_v3.py:1144`):** `use_ep = num_expert_parallelism>1 AND total_TP==1` (`total_TP = tp_size * attn_dp_size`). **EP and TP/attn-DP are mutually exclusive** in the existing path. The Phase-1b 8-chip gate must pick one mode for the routed experts (pure-EP over `expert_axis_name`, or TP via GMM_TP) — **state which, since the gate explicitly exercises `expert_axis_name` sharding.**
- **Stress-fixture caveat (reconcile §6 Phase 1b / §8):** `sparse_moe.py` uses `local_expert_size = num_local_experts // num_expert_parallelism` (integer division) and reshapes to `(EP, EP, local_expert_size)`. A shard that truly owns **0 experts** gives `local_expert_size==0` and a degenerate `ragged_all_to_all`/local_permute that **faults**, not silently arms S1. So the stress fixture must be **uneven for the TP/reshard tiling** (experts not a multiple of the TP/`MOE_TENSOR` axis; idle data-shard rows) to arm the uninit-HBM-on-reshard class — **NOT an empty EP expert-shard** the `//` partition cannot represent. If an actually-empty EP shard is wanted, the current EP path does not support `local_expert_size==0` and must be padded, or the fixture must run in TP mode.

### New DSA tensors — explicit specs (default to replicated otherwise, which is wrong on the token dim and a long-context memory cost)
- **Indexer projections** (`GlmMoeDsaIndexer`: `wq_b`, `wk`, `weights_proj`, `k_norm`) — no inherited answer: shard `wq_b`/`wk` outputs on `ATTN_HEAD` (per-head, matching the indexer's per-head `weights_proj` and head-sum); keep `k_norm` (LayerNorm) **replicated**; run through the Phase-1b N-dev==1-dev fp32 + NaN-poison gate like the rest.
- **Indexer-key KV cache** `[tokens, index_head_dim=128]` (single index head): **register as a second KV cache group**, mirroring the GPU reference — vLLM's `DeepseekV32IndexerCache` is an `AttentionLayerBase` whose `get_kv_cache_spec` returns `MLAAttentionSpec(num_kv_heads=1, head_size=128)`. On TPU, emit a single-head MLA-like spec via the existing `_create_attention_spec` / `get_attention_page_size_bytes` path (`kv_cache.py:155`) so it flows through the multi-group machinery already used for hybrid mamba+attention (`_hybrid_uniform_page_size_bytes`, per-`gid` iteration over `kv_cache_groups`, `MultiGroupBlockTable`). The MLA latent cache (576 numbers/token) and the indexer cache (128 keys/token) have different `page_size_bytes`, so they are **naturally two groups** (v1 may share allocation; state the choice). Sharding spec: mirror the latent cache — `P(BATCH, None, ATTN_HEAD)` (the single head makes `ATTN_HEAD` degenerate ≈ replicated, matching MQA). ⚠️ **S1 watch:** the V4 fork hit decode-collapse where indexer/compressor KV state on EMPTY attn_dp token shards seeded uninitialized HBM into replicated decode state (`deepseek_v4_attention.py:542-547`); **host-stack into the sharded layout and keep empty-shard zeroing — do not device-reshard it.**
- **`topk_indices`** `[tokens, index_topk]` int32 (`-1` sentinel) in `AttentionMetadata`: shard on the **token axis** so it rides q into the sparse kernel's `shard_map` with no extra gather — `P(ATTN_DATA, None)` (`ATTN_DATA = ('data','attn_dp','attn_dp_expert')`), mirroring the fork's `_sparse_attn_kernel_sharded` in_specs (`topk_idxs` `P(None, td, None)`, q `P(None, td, None, None)`). It must be added to `AttentionMetadata`'s `register_dataclass` `data_fields` to be a sharded leaf. **Coordinate space: per-request-local (0..seq_len-1)** — see §11 Phase 4; the kernel demuxes per `seq_idx` via `cu_kv_lens` + `block_table`, so a selected index never crosses a request boundary.

### Scope
v1 (Phase 1b) inherits all the above and gates degrees per-run on the 8-chip mesh. **Production TP/EP degree tuning and multi-host** are Phase 3/§14.

## 13. Performance & fp8 quantization roadmap (production)

§10 covers the correctness-era idiom (kernels performant-from-day-1, bounded human-triggered tuning). This section is the **production** perf/quant arc, with measurable gates, consumed by §11 Phase 6. All gates keep the §10 guardrails (full correctness suite green; tuned-vs-baseline ≤5% regression; frozen kernel commit; human-reviewed apply-back).

### Perf-critical: the DSA sparse-gather rewrite (Phase 2→Phase 5 item, own gate)
The ported fork kernel's one-hot gather is **O(N) in total context length, not O(index_topk)** — `onehot[K,N] @ kv[N,D]` reads every resident KV row. It is a **correctness-only / short-context bridge**. At production long context (the point of DSA) the all-N read dominates and the sparsity win is lost. The production gather **must** switch to the **gen-6 DMA scalar-prefetch idiom** already proven in-repo (`mla/v2/kernel.py:2454` PrefetchScalarGridSpec/`num_scalar_prefetch`, `:661` make_async_copy; `ragged_paged_attention/v3`): SMEM scalar-prefetch the per-request-local index array, kv as an HBM operand, per-row `make_async_copy` in a statically-unrolled loop, size-0 copy for the `-1` sentinel — softmax math unchanged, only the kvg-construction changes.
**Gate (own, separate from correctness):** the §10 structural smoke (revised) sweeps context length N at real `index_topk=2048` — at least N≈2048 (all-N==topk) and N at a representative production length (N≫topk) — to size when the rewrite becomes mandatory while it is still cheap. Acceptance: rewritten kernel's decode-step latency scales with `index_topk`, not N (measured: ≥2x speedup at N=8×topk vs the one-hot bridge, no correctness regression vs the jnp-ref).

### GLM-shaped kernel tuning (mandatory for any perf claim, not optional polish)
**Tuned-param reuse is a no-op for GLM v1:** every entry in `tuned_params.py` is keyed on `actual_num_q_heads=128` + fp8 `q_dtype`/`kv_dtype` (DeepSeek-V3 decode shapes). GLM is 64 heads + bf16 (v1), so the `TuningKey` never matches and `get_tuned_params` (`:379`) silently falls to hardcoded defaults (`num_kv_pages_per_block=3, num_queries_per_block=1, decode_batch_size=4`). Before any GLM perf claim, **add GLM-shaped keys to `tuned_params_mapping`** (`actual_num_q_heads=64`, `actual_lkv_dim=512`, `actual_r_dim=64`, real `page_size`/`kv_packing`, and the dtype matching whichever precision is shipped), then run the §10 bounded sweep at real GLM dims (tiny-config perf is meaningless — the lookup is exact-shape-keyed). Also write the dedicated `sparse_attn` tuner (subclass `KernelTunerBase`).

### fp8 quantization (Phase 6 — gated behind batching, own correctness ladder)
v1 is bf16/fp32; the GPU DSA path is fp8 (DeepGEMM `fp8_mqa_logits`, fp8 indexer). The production indexer should mirror HF's **fp32-pinned scoring/weighting** (§9 #9: `weights_proj` fp32, `relu(q.float()@k.float())*scale`, head-sum in fp32) and downcast only after top-k — fp8 applies to the *gather/logits* path, not the reduction that feeds top-k. Reconcile against the vLLM fp8 indexer (ReLU location inside compiled deep_gemm — not readable Python; confirm empirically, §9 #4). fp8 weight quant (UE8M0) ties into the existing quant config path; the absorbed MLA already has an fp8 kernel mode.
**Ladder:** single-chip fp8-vs-bf16 jnp-ref delta within a measured fp8 floor → 8-chip N-dev==1-dev → throughput/memory win quantified vs bf16. fp8 ships only when its correctness ladder is green; no fp8 perf number without it.

### Measurable production perf gates (summary)
- DSA gather: decode latency scales with `index_topk`, not N (≥2x at N=8×topk vs one-hot bridge).
- MLA + DSA tuned at real GLM dims (64 heads, real page_size): tuned-vs-baseline ≤5% regression test green on GLM/DSA shapes.
- Batched decode (§11 Phase 4): throughput scales ~linearly with `max_num_seqs` up to the mesh's compute roofline, value-invariant per request.
- fp8 (§11 Phase 6): memory/throughput win quantified vs bf16 with correctness ladder green.
